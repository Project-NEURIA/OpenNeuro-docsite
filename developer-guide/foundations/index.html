Theoretical Foundations

Idea

OpenNeuro models real-time data processing as a network of composable components with typed interfaces. This page develops the categorical semantics of this system incrementally.

1. Categories

Definition 1.1 (Category)

A category C consists of:

  1. A collection of objects Ob(C)\text{Ob}(\mathbf{C})Ob(C).
  2. For every pair of objects A,BA, BA,B, a collection of morphisms Hom(A,B)\text{Hom}(A, B)Hom(A,B).
  3. For every object AAA, an identity morphism idA∈Hom(A,A)\text{id}_A \in \text{Hom}(A, A)idA​∈Hom(A,A).
  4. For every triple A,B,CA, B, CA,B,C and morphisms f∈Hom(A,B)f \in \text{Hom}(A, B)f∈Hom(A,B), g∈Hom(B,C)g \in \text{Hom}(B, C)g∈Hom(B,C), a composition g∘f∈Hom(A,C)g \circ f \in \text{Hom}(A, C)g∘f∈Hom(A,C).

subject to:

  • Associativity. For f:A→Bf : A \to Bf:A→B, g:B→Cg : B \to Cg:B→C, h:C→Dh : C \to Dh:C→D:

h∘(g∘f)=(h∘g)∘fh \circ (g \circ f) = (h \circ g) \circ fh∘(g∘f)=(h∘g)∘f

  • Identity. For f:A→Bf : A \to Bf:A→B:

f∘idA=f=idB∘ff \circ \text{id}_A = f = \text{id}_B \circ ff∘idA​=f=idB​∘f

Roughly, in Python:

class Category[A, B](Protocol):
    @staticmethod
    def id[X]() -> Category[X, X]: ...
    def compose[C](self, other: Category[B, C]) -> Category[A, C]: ...
    # self.compose(other.compose(h)) == self.compose(other).compose(h)
    # id().compose(f) == f == f.compose(id())

Remark 1.2

A category is the minimal structure needed to talk about “things that compose.” This is required to describe the syntax of OpenNeuro’s graphical editor, since components can compose to form graphs.

Example 1.3 (The category of frame types and components)

We construct a simple category Comp whose:

  • Objects are frame types: Audio\text{Audio}Audio, Text\text{Text}Text, Video\text{Video}Video, Interrupt\text{Interrupt}Interrupt, BodyPose\text{BodyPose}BodyPose.
  • Morphisms are components. Each component fff has a typed interface f:A→Bf : A \to Bf:A→B, meaning it receives frames of type AAA and produces frames of type BBB. The morphisms of Comp are the following components:

ASR:Audio→Text\text{ASR} : \text{Audio} \to \text{Text}ASR:Audio→Text TTS:Text→Audio\text{TTS} : \text{Text} \to \text{Audio}TTS:Text→Audio LLM:Text→Text\text{LLM} : \text{Text} \to \text{Text}LLM:Text→Text

the identity (passthrough) morphisms for each object, e.g. idAudio:Audio→Audio\text{id}_{\text{Audio}} : \text{Audio} \to \text{Audio}idAudio​:Audio→Audio, and anything that can be formed by composing these.

Composition is wiring: given ASR:Audio→Text\text{ASR} : \text{Audio} \to \text{Text}ASR:Audio→Text and LLM:Text→Text\text{LLM} : \text{Text} \to \text{Text}LLM:Text→Text, their composition is:

LLM∘ASR:Audio→Text\text{LLM} \circ \text{ASR} : \text{Audio} \to \text{Text}LLM∘ASR:Audio→Text

Definition 1.4 (Free category)

Given a directed graph GGG with nodes (objects) and edges (primitive morphisms), the free category F(G)F(G)F(G) is the category whose morphisms are all finite directed paths in GGG, composed by path concatenation, with the empty path as identity.

No equations are imposed beyond the category axioms. Two morphisms are equal if and only if they are the same sequence of primitive morphisms. This means every distinct way of composing primitives produces a distinct morphism — the free category is the space of all programs that can be constructed from the given primitives.

See nLab: free category .

Remark 1.5

Comp as constructed in Example 1.3 is a free category, where the primitive morphisms are the (primitive) components and the objects are the frame types.

Remark 1.6 (Limitation)

Consider a graph where ASR:Audio→Text\text{ASR} : \text{Audio} \to \text{Text}ASR:Audio→Text and PoseRenderer:BodyPose→Video\text{PoseRenderer} : \text{BodyPose} \to \text{Video}PoseRenderer:BodyPose→Video run side by side with no connection between them. This cannot be expressed in Comp — composition (∘\circ∘) requires the output of one morphism to match the input of the next. There is no operation for placing two independent morphisms in parallel. To model this, we need additional structure.

2. Commutative monoidal categories

Definition 2.1 (Monoidal category)

A monoidal category (C,⊗,I)(\mathbf{C}, \otimes, I)(C,⊗,I) is a category C\mathbf{C}C equipped with:

  1. A bifunctor ⊗:C×C→C\otimes : \mathbf{C} \times \mathbf{C} \to \mathbf{C}⊗:C×C→C, called the tensor product or monoidal product.
  2. A unit object III.
  3. Natural isomorphisms:
    • Associator: αA,B,C:(A⊗B)⊗C→∼A⊗(B⊗C)\alpha_{A,B,C} : (A \otimes B) \otimes C \xrightarrow{\sim} A \otimes (B \otimes C)αA,B,C​:(A⊗B)⊗C∼​A⊗(B⊗C)
    • Left unitor: λA:I⊗A→∼A\lambda_A : I \otimes A \xrightarrow{\sim} AλA​:I⊗A∼​A
    • Right unitor: ρA:A⊗I→∼A\rho_A : A \otimes I \xrightarrow{\sim} AρA​:A⊗I∼​A

satisfying the triangle and pentagon coherence conditions (see nLab: monoidal category ).

Definition 2.2 (Symmetric monoidal category)

A symmetric monoidal category is a monoidal category equipped with a natural isomorphism (the braiding):

σA,B:A⊗B→∼B⊗A\sigma_{A,B} : A \otimes B \xrightarrow{\sim} B \otimes AσA,B​:A⊗B∼​B⊗A

such that σB,A∘σA,B=idA⊗B\sigma_{B,A} \circ \sigma_{A,B} = \text{id}_{A \otimes B}σB,A​∘σA,B​=idA⊗B​.

Definition 2.3 (Commutative monoidal category)

A monoidal category is commutative (also called strictly symmetric) when the braiding is the identity:

A⊗B=B⊗AA \otimes B = B \otimes AA⊗B=B⊗A

That is, the tensor product is commutative on the nose, not merely up to isomorphism.

Remark 2.4

A commutative monoidal category is a symmetric monoidal category where the symmetry is trivial. This is a stronger condition than mere symmetry.

Remark 2.5 (Interfaces)

In the simple Comp of §1, objects are bare frame types like Audio\text{Audio}Audio or Text\text{Text}Text. In practice, components often have multiple input or output ports. For example, an LLM component takes both text and an interrupt signal:

LLM:(text:Text,  interrupt:Interrupt)→Text\text{LLM} : (\text{text}: \text{Text},\; \text{interrupt}: \text{Interrupt}) \to \text{Text}LLM:(text:Text,interrupt:Interrupt)→Text

We therefore refine our notion of object: objects in Comp are interfaces — named records of typed ports. A single-port interface like Audio\text{Audio}Audio is just a record with one field. Multi-port interfaces arise naturally from the components themselves, not only from parallel composition.

Remark 2.6 (Parallel composition in Comp)

In OpenNeuro, the tensor product is commutative on the nose at both levels:

  • Objects. Component interfaces use named ports (Python NamedTuple fields), not positionally ordered tuples. The interface (audio: Audio, video: Video) is the same object as (video: Video, audio: Audio) — the names determine identity, not the order. So A⊗B=B⊗AA \otimes B = B \otimes AA⊗B=B⊗A.

  • Morphisms. Parallel execution has no notion of “first” or “second” — both components run independently on their own threads. So f⊗g=g⊗ff \otimes g = g \otimes ff⊗g=g⊗f.

This makes Comp a commutative monoidal category with:

  • A⊗BA \otimes BA⊗B = the combined interface with all ports from AAA and BBB.
  • III = the empty interface (a component with no inputs or no outputs).

Example 2.7 (Parallel composition in Comp)

Recall from Remark 1.6 that ASR\text{ASR}ASR and PoseRenderer\text{PoseRenderer}PoseRenderer running side by side could not be expressed in a plain category. With the monoidal product, we can now write:

ASR⊗PoseRenderer:Audio⊗BodyPose→Text⊗Video\text{ASR} \otimes \text{PoseRenderer} : \text{Audio} \otimes \text{BodyPose} \to \text{Text} \otimes \text{Video}ASR⊗PoseRenderer:Audio⊗BodyPose→Text⊗Video

Both components run in parallel. The composite interface has two input ports (Audio\text{Audio}Audio, BodyPose\text{BodyPose}BodyPose) and two output ports (Text\text{Text}Text, Video\text{Video}Video).

Remark 2.8 (Relation to Haskell’s Arrow)

The Arrow type class in Haskell corresponds closely to a (strict) monoidal category. The operations are:

Haskell ArrowCategorical

>>>sequential composition (∘\circ∘)
***parallel composition (⊗\otimes⊗)

See nLab: Arrow .

Remark 2.9 (Limitation)

Consider an AgentState:Text→History\text{AgentState} : \text{Text} \to \text{History}AgentState:Text→History component that accumulates conversation history, and an LLM:History→Text\text{LLM} : \text{History} \to \text{Text}LLM:History→Text component. Composing them sequentially gives:

LLM∘AgentState:Text→Text\text{LLM} \circ \text{AgentState} : \text{Text} \to \text{Text}LLM∘AgentState:Text→Text

But we want the Text\text{Text}Text output of LLM\text{LLM}LLM to feed back into AgentState\text{AgentState}AgentState, forming a loop — the agent responds, the response is appended to history, and the updated history feeds the next LLM call. A commutative monoidal category has no operation for this — there is no way to wire an output back into an input. To model feedback, we need trace.

3. Traced commutative monoidal categories

Definition 3.1 (Trace)

Let (C,⊗,I)(\mathbf{C}, \otimes, I)(C,⊗,I) be a symmetric monoidal category. A trace on C\mathbf{C}C is a family of functions:

TrA,BX:Hom(A⊗X,  B⊗X)→Hom(A,B)\text{Tr}^X_{A,B} : \text{Hom}(A \otimes X, \; B \otimes X) \to \text{Hom}(A, B)TrA,BX​:Hom(A⊗X,B⊗X)→Hom(A,B)

for every triple of objects A,B,XA, B, XA,B,X, satisfying naturality and coherence axioms (see nLab: traced monoidal category ).

Remark 3.2

Intuitively, TrA,BX\text{Tr}^X_{A,B}TrA,BX​ takes a morphism that has an “extra” input and output of type XXX and produces a morphism where XXX is wired internally as a feedback loop. The external interface shrinks from (A⊗X)→(B⊗X)(A \otimes X) \to (B \otimes X)(A⊗X)→(B⊗X) to just A→BA \to BA→B.

Example 3.3 (Feedback in Comp)

Recall from Remark 2.9 the AgentState:Text→History\text{AgentState} : \text{Text} \to \text{History}AgentState:Text→History and LLM:History→Text\text{LLM} : \text{History} \to \text{Text}LLM:History→Text components. Their composition LLM∘AgentState:Text→Text\text{LLM} \circ \text{AgentState} : \text{Text} \to \text{Text}LLM∘AgentState:Text→Text is a morphism with matching input and output types. We can model this as a morphism with an extra feedback channel:

P:Text⊗Text→Text⊗TextP : \text{Text} \otimes \text{Text} \to \text{Text} \otimes \text{Text}P:Text⊗Text→Text⊗Text

Applying trace over Text\text{Text}Text:

TrText(P):Text→Text\text{Tr}^{\text{Text}}(P) : \text{Text} \to \text{Text}TrText(P):Text→Text

The feedback channel becomes internal — the LLM’s text output feeds back into AgentState, whose history output feeds the next LLM call. From the outside, it is simply a Text→Text\text{Text} \to \text{Text}Text→Text component.

Remark 3.4 (Relation to Haskell’s ArrowLoop)

The ArrowLoop type class adds feedback to Arrow:

class Arrow a => ArrowLoop a where
    loop :: a (b, d) (c, d) -> a b c

This is exactly the trace operator: loop takes a morphism of type (B⊗D)→(C⊗D)(B \otimes D) \to (C \otimes D)(B⊗D)→(C⊗D) and produces one of type B→CB \to CB→C by feeding DDD back. The rec keyword in arrow notation is syntactic sugar for loop.

Remark 3.5 (Limitation)

Consider an ASR:Audio→Text\text{ASR} : \text{Audio} \to \text{Text}ASR:Audio→Text whose text output should feed both an LLM\text{LLM}LLM and a Logger\text{Logger}Logger simultaneously. Or two independent ASR\text{ASR}ASR components whose text outputs should merge into a single LLM\text{LLM}LLM input. A traced commutative monoidal category has no operation for either — every output goes to exactly one input. We need:

  • Fan-out (copying): one output feeds multiple downstream components.
  • Fan-in (merging): multiple outputs feed into one input.
  • Discard: an output that goes nowhere.

4. Copy, merge, and discard

Definition 4.1 (Commutative comonoid)

Let (C,⊗,I)(\mathbf{C}, \otimes, I)(C,⊗,I) be a monoidal category. A commutative comonoid on an object AAA is a pair of morphisms:

ΔA:A→A⊗A(copy)\Delta_A : A \to A \otimes A \quad \text{(copy)}ΔA​:A→A⊗A(copy) εA:A→I(discard)\varepsilon_A : A \to I \quad \text{(discard)}εA​:A→I(discard)

satisfying coassociativity, counitality, and commutativity (see nLab: comonoid ).

Definition 4.2 (Commutative monoid)

Dually, a commutative monoid on an object AAA is a pair of morphisms:

∇A:A⊗A→A(merge)\nabla_A : A \otimes A \to A \quad \text{(merge)}∇A​:A⊗A→A(merge) ηA:I→A(create)\eta_A : I \to A \quad \text{(create)}ηA​:I→A(create)

satisfying associativity, unitality, and commutativity (see nLab: monoid ).

Definition 4.3 (gs-monoidal category)

A gs-monoidal category (garbage-sharing monoidal category) is a symmetric monoidal category in which every object is equipped with a commutative comonoid structure (ΔA,εA)(\Delta_A, \varepsilon_A)(ΔA​,εA​) that is natural in AAA.

“Garbage” refers to discard (ε\varepsilonε), and “sharing” refers to copy (Δ\DeltaΔ).

See [Corradini & Gadducci, 1999] for the original definition.

Definition 4.4 (gs-monoidal category with merges)

We extend the gs-monoidal structure by additionally equipping every object with a natural commutative monoid structure (∇A,ηA)(\nabla_A, \eta_A)(∇A​,ηA​). This gives every object the structure of a commutative bimonoid: both a commutative monoid and a commutative comonoid.

Example 4.5 (Fan-out, fan-in, and discard in Comp)

In OpenNeuro, these operations are provided by the channel system:

  • Copy (Δ\DeltaΔ). A Sender holds a list of Channels. When it sends a frame, every channel receives a copy. This is fan-out: one output going to multiple downstream components.

  • Discard (ε\varepsilonε). A Sender with no connected channels. The frame is produced but nobody reads it. The data is silently discarded.

  • Merge (∇\nabla∇). Multiple Senders connected to the same Channel. Frames from different upstream components are interleaved by arrival order into a single stream.

  • Create (η\etaη). A source component (e.g. Microphone) that produces frames from nothing — it has no input channels, only outputs.

These operations are inherent to the channel design, not bolted on. Any component’s output can fan out to arbitrarily many downstream components, and any input can receive from arbitrarily many upstream components, without any special wiring logic.

5. The type lattice

Definition 5.1 (Partial order on objects)

The objects of Comp carry a partial order ≤\leq≤ given by the subtyping relation on frame types. We write A≤BA \leq BA≤B when AAA is a subtype of BBB.

In OpenNeuro, this is the class hierarchy:

            Frame
          / |  |  \
   Audio  Text  Video  BodyPose  ...
           |
          EOS

So EOS≤Text≤Frame\text{EOS} \leq \text{Text} \leq \text{Frame}EOS≤Text≤Frame, but Audio\text{Audio}Audio and Video\text{Video}Video are incomparable.

Remark 5.2 (Wiring compatibility)

A connection from a Sender[A]\text{Sender}[A]Sender[A] to a Receiver[B]\text{Receiver}[B]Receiver[B] is valid when A≤BA \leq BA≤B. This means a component that outputs EOS\text{EOS}EOS frames can wire into any input that accepts Text\text{Text}Text, because EOS≤Text\text{EOS} \leq \text{Text}EOS≤Text.

The partial order gives rise to implicit coercion morphisms ιA,B:A→B\iota_{A,B} : A \to BιA,B​:A→B for every A≤BA \leq BA≤B. These are the “silent” type conversions that the system handles without the user writing an explicit adapter component.

Remark 5.3

In categorical terms, the partial order makes the collection of objects a poset , and the coercion morphisms are the unique morphisms guaranteed by the partial order. This enriches our category with a subtyping discipline: composition respects the order, and the monoidal product extends it componentwise.

6. Higher-order components

Definition 6.1 (Functor)

A functor F:C→DF : \mathbf{C} \to \mathbf{D}F:C→D between two categories consists of:

  1. A mapping on objects: A↦F(A)A \mapsto F(A)A↦F(A).
  2. A mapping on morphisms: (f:A→B)↦(F(f):F(A)→F(B))(f : A \to B) \mapsto (F(f) : F(A) \to F(B))(f:A→B)↦(F(f):F(A)→F(B)).

preserving identity and composition:

F(idA)=idF(A)F(\text{id}_A) = \text{id}_{F(A)}F(idA​)=idF(A)​ F(g∘f)=F(g)∘F(f)F(g \circ f) = F(g) \circ F(f)F(g∘f)=F(g)∘F(f)

An endofunctor is a functor from a category to itself: F:C→CF : \mathbf{C} \to \mathbf{C}F:C→C.

See nLab: functor .

Example 6.2 (Generic components as endofunctors)

A component that is generic over its frame type defines an endofunctor on Comp. For instance, a Passthrough[T] component — parameterized by a type TTT — maps:

  • Each object TTT to itself.
  • Each morphism f:A→Bf : A \to Bf:A→B to a morphism Passthrough(f):A→B\text{Passthrough}(f) : A \to BPassthrough(f):A→B that applies fff while passing data through.

More practical examples of endofunctors on Comp include:

  • A profiler that wraps any component and adds timing measurements, without changing its interface type.
  • A retry wrapper that takes any component f:A→Bf : A \to Bf:A→B and produces a fault-tolerant version Retry(f):A→B\text{Retry}(f) : A \to BRetry(f):A→B.

These are structure-preserving: wrapping a composition is the same as composing the wrapped versions. That is precisely the functor law F(g∘f)=F(g)∘F(f)F(g \circ f) = F(g) \circ F(f)F(g∘f)=F(g)∘F(f).

7. Subgraphs and named composition

Definition 7.1 (Graph expression language)

We define an expression language for morphisms in the free category. An expression eee has the following grammar:

e    ::=    prim(c)  ∣  e1∘e2  ∣  e1⊗e2  ∣  name(x)e \;\;::=\;\; \text{prim}(c) \;\mid\; e_1 \circ e_2 \;\mid\; e_1 \otimes e_2 \;\mid\; \text{name}(x)e::=prim(c)∣e1​∘e2​∣e1​⊗e2​∣name(x)

where ccc ranges over primitive components (generators), and xxx ranges over names drawn from an environment Γ:Name→Value\Gamma : \text{Name} \to \text{Value}Γ:Name→Value, where values are morphisms in the free category.

Definition 7.2 (Operational semantics)

The evaluation judgment Γ⊢e⇓v\Gamma \vdash e \Downarrow vΓ⊢e⇓v maps an expression eee under environment Γ\GammaΓ to a morphism vvv in F(G)F(G)F(G):

Γ⊢prim(c)⇓c\frac{}{\Gamma \vdash \text{prim}(c) \Downarrow c}Γ⊢prim(c)⇓c​

Γ⊢e1⇓fΓ⊢e2⇓gΓ⊢e1∘e2⇓f∘g\frac{\Gamma \vdash e_1 \Downarrow f \quad \Gamma \vdash e_2 \Downarrow g}{\Gamma \vdash e_1 \circ e_2 \Downarrow f \circ g}Γ⊢e1​∘e2​⇓f∘gΓ⊢e1​⇓fΓ⊢e2​⇓g​

Γ⊢e1⇓fΓ⊢e2⇓gΓ⊢e1⊗e2⇓f⊗g\frac{\Gamma \vdash e_1 \Downarrow f \quad \Gamma \vdash e_2 \Downarrow g}{\Gamma \vdash e_1 \otimes e_2 \Downarrow f \otimes g}Γ⊢e1​⊗e2​⇓f⊗gΓ⊢e1​⇓fΓ⊢e2​⇓g​

Γ(x)=vΓ⊢name(x)⇓v\frac{\Gamma(x) = v}{\Gamma \vdash \text{name}(x) \Downarrow v}Γ⊢name(x)⇓vΓ(x)=v​

Remark 7.3 (Grouping as environment extension)

When a user selects a subgraph and groups it under a name xxx, this corresponds to extending the environment:

Γ′=Γ[x↦v]\Gamma' = \Gamma[x \mapsto v]Γ′=Γ[x↦v]

where vvv is the morphism obtained by evaluating the selected subexpression. For example, given a graph with ASR:Audio→Text\text{ASR} : \text{Audio} \to \text{Text}ASR:Audio→Text and LLM:Text→Text\text{LLM} : \text{Text} \to \text{Text}LLM:Text→Text, the expression is:

e=LLM∘ASRe = \text{LLM} \circ \text{ASR}e=LLM∘ASR

Grouping under the name “understanding” produces:

Γ′=Γ[understanding↦LLM∘ASR]\Gamma' = \Gamma[\text{understanding} \mapsto \text{LLM} \circ \text{ASR}]Γ′=Γ[understanding↦LLM∘ASR]

The graph can now be written as name(understanding)\text{name}(\text{understanding})name(understanding), which evaluates to the same morphism. The grouping operation does not change the denoted morphism — it introduces a name for a composite, making it referenceable.

Remark 7.4 (Ungrouping)

Ungrouping is the inverse: given Γ′=Γ[x↦v]\Gamma' = \Gamma[x \mapsto v]Γ′=Γ[x↦v], ungrouping xxx removes the binding and inlines vvv back into the expression, recovering the original environment Γ\GammaΓ.

Definition 7.5 (Open and closed expressions)

An expression eee is closed under environment Γ\GammaΓ when every name(x)\text{name}(x)name(x) occurring in eee satisfies x∈dom(Γ)x \in \text{dom}(\Gamma)x∈dom(Γ). A closed expression can be fully evaluated by the rules of Definition 7.2.

An expression is open when it contains free names — name(x)\text{name}(x)name(x) where x∉dom(Γ)x \notin \text{dom}(\Gamma)x∈/dom(Γ). Evaluation gets stuck at any free name since no rule applies.

Remark 7.6 (Abstract components)

An open expression contains abstract components: it has a well-defined typed interface (its input and output types can be inferred from the structure), but it cannot be executed because some subexpressions are unresolved.

To execute an abstract graph, the user must supply a substitution σ:FreeNames(e)→Value\sigma : \text{FreeNames}(e) \to \text{Value}σ:FreeNames(e)→Value mapping each free name to a concrete morphism of matching type. Applying the substitution closes the expression:

Γ′=Γ∪σ\Gamma' = \Gamma \cup \sigmaΓ′=Γ∪σ

Γ′⊢e⇓v\Gamma' \vdash e \Downarrow vΓ′⊢e⇓v

8. Putting it together

Remark 8.1

Extending the free category of Definition 1.4 with all the structure introduced in §2–§4, OpenNeuro’s graph language generates a free traced commutative gs-monoidal category with merges — the free category built from the declared components as primitive morphisms, closed under sequential composition (∘\circ∘), parallel composition (⊗\otimes⊗), trace (Tr\text{Tr}Tr), copy (Δ\DeltaΔ), merge (∇\nabla∇), and discard (ε\varepsilonε), with no additional equations.

This free category is the semantic domain of the expression language defined in §7. Every closed expression evaluates to a morphism in this category. The expressions themselves — which may contain named subgraphs via name(x)\text{name}(x)name(x) — are the syntax that users work with in the editor. The operational semantics of §7 bridges the two: it flattens named compositions into morphisms of the free category.

Definition 8.2 (Interpretation)

An interpretation of the free category is a structure-preserving functor:

⟦−⟧:F(Generators)→Sem\llbracket - \rrbracket : F(\text{Generators}) \to \mathbf{Sem}[[−]]:F(Generators)→Sem

from the free category into a semantic category Sem\mathbf{Sem}Sem where morphisms have computational meaning (e.g., actual running threads passing real frames through channels).

The interpretation functor must preserve all structure:

  • ⟦g∘f⟧=⟦g⟧∘⟦f⟧\llbracket g \circ f \rrbracket = \llbracket g \rrbracket \circ \llbracket f \rrbracket[[g∘f]]=[[g]]∘[[f]]
  • ⟦f⊗g⟧=⟦f⟧⊗⟦g⟧\llbracket f \otimes g \rrbracket = \llbracket f \rrbracket \otimes \llbracket g \rrbracket[[f⊗g]]=[[f]]⊗[[g]]
  • ⟦TrX(f)⟧=Tr⟦X⟧(⟦f⟧)\llbracket \text{Tr}^X(f) \rrbracket = \text{Tr}^{\llbracket X \rrbracket}(\llbracket f \rrbracket)[[TrX(f)]]=Tr[[X]]([[f]])

Because the source is free, the interpretation is uniquely determined by its action on the generators. Once you assign a concrete implementation to each primitive component, the semantics of every composite graph is fixed automatically by functoriality.

Remark 8.3 (Abstract components)

A generator in the free category that has not been assigned an interpretation is an abstract component — declared with a typed interface but no implementation. A graph containing abstract components cannot be fully interpreted (executed) until every abstract generator is given a concrete meaning. This is the categorical account of the open expressions described in Definition 7.5: an unbound name and an uninterpreted generator are two views of the same concept — a typed placeholder that must be filled in before the graph can run.
